# โครงสร้างโปรเจกต์นี้ (สำหรับเรียนรู้ Frontend)

โน้ตอธิบายว่าทำไมโค้ดถึงถูกแยกไว้คนละโฟลเดอร์ และเวลาจะเขียนโค้ดใหม่ ควรเอาไปวางไว้ตรงไหน

## หลักการรวมๆ

เว็บหนึ่งหน้า จริงๆ แล้วประกอบจากของ 3 อย่าง:

1. **หน้าตา/ตำแหน่งข้อมูล** — ผู้ใช้เห็นอะไร กดอะไรได้บ้าง
2. **ชิ้นส่วนที่ใช้ซ้ำได้** — ปุ่ม, การ์ด, ช่องกรอกข้อมูล ที่หน้าไหนก็เรียกใช้เหมือนกัน
3. **ตรรกะ/ข้อมูล** — คำนวณอะไร, ไปเอาข้อมูลจากไหน, เก็บไว้ที่ไหน

โปรเจกต์นี้แยก 3 อย่างนี้เป็นคนละโฟลเดอร์ เพื่อไม่ให้โค้ดปนกันจนงง

## `app/` — แต่ละหน้าเว็บ

Next.js กำหนดว่า 1 โฟลเดอร์ = 1 หน้าเว็บ (1 URL) โดยดูจากชื่อโฟลเดอร์ตรงๆ:

```
app/page.tsx              → เปิดเว็บที่ "/"           (หน้า Home)
app/login/page.tsx        → "/login"                 (หน้า login)
app/expenses/page.tsx     → "/expenses"               (หน้ารายการ)
app/expenses/new/page.tsx → "/expenses/new"           (หน้าเพิ่มรายจ่าย)
app/expenses/[id]/page.tsx→ "/expenses/xxxx"          ([id] = ตัวแปรในทาง URL)
```

ในไฟล์ `page.tsx` แต่ละหน้า จะเป็นโค้ดที่ "ประกอบร่าง" หน้านั้นๆ — เอาชิ้นส่วนจาก `components/` มาต่อกัน แล้วดึงข้อมูลจาก `store/` มาแสดง เป็นจุดที่ตอบว่า **"หน้านี้หน้าตาเป็นยังไง"**

ในนี้ยังมี `app/api/.../route.ts` — อันนี้ไม่ใช่หน้าเว็บที่คนเห็น แต่เป็น "ประตูหลังบ้าน" ที่ฝั่งเบราว์เซอร์ยิง request มาขอ/ส่งข้อมูล เช่น กดปุ่มบันทึก → เว็บส่งข้อมูลมาที่ `app/api/expenses/route.ts` → โค้ดตรงนี้ค่อยไปเขียนลง Google Sheet จริงๆ

## `components/` — ชิ้นส่วน UI ที่ใช้ซ้ำ

กติกา: **ถ้าอะไรถูกใช้มากกว่า 1 หน้า ให้แยกออกมาเป็น component** จะได้ไม่ต้องเขียนโค้ดหน้าตาเดิมซ้ำๆ ทุกหน้า

- `components/ui/` — ของเล็กๆ ทั่วไป: ปุ่ม (`Button`), ช่องกรอก (`Input`), การ์ด (`Card`), ไอคอนวงกลมหมวดหมู่ (`Avatar`) — พวกนี้ไม่รู้ด้วยซ้ำว่าตัวเองอยู่หน้าไหน แค่รับค่าเข้ามาแล้ววาดออกไป
- `components/layout/` — โครงหน้าที่ซ้ำทุกหน้า: แถบบนสุด (`TopBar`), เมนูล่าง (`BottomNav`), กรอบทั้งหน้า (`Screen`)
- `components/home/` — ชิ้นส่วนที่ใช้เฉพาะหน้า Home เท่านั้น (`WeekChart` กราฟแท่ง 7 วัน) — จับใส่โฟลเดอร์ย่อยตามหน้าที่ใช้ ไม่ต้องไปปนกับ `ui/`

**เวลาจะรู้ว่าควรแยกเป็น component ไหม**: ลองถามตัวเองว่า "ถ้าฉันก็อปปี้ JSX นี้ไปวางอีกหน้า จะเหมือนเดิมไหม" ถ้าใช่ → แยกออกมา

## `lib/` — ฟังก์ชันช่วยที่ไม่เกี่ยวกับหน้าตา

โค้ดในนี้ **ไม่มี JSX ไม่มีปุ่ม ไม่มีสี** — เป็นแค่ฟังก์ชันรับ input คืน output ล้วนๆ:

- `formatters.ts` — แปลงตัวเลข/วันที่ให้อ่านง่าย เช่น 1500 → "฿1,500"
- `validation.ts` — เช็คว่าข้อมูลที่กรอกมาถูกต้องไหม (จำนวนเงินต้อง > 0 ฯลฯ)
- `categories.ts` — รายชื่อหมวดหมู่รายจ่าย พร้อมสีและตัวอักษรของแต่ละหมวด
- `sheets.ts` — ฟังก์ชันคุยกับ Google Sheet (อ่าน/เขียน/ลบแถว)
- `auth.ts` — เช็ครหัสผ่าน, ออก/ตรวจ token ล็อกอิน
- `utils.ts` — ของเล็กจิปาถะที่ใช้ทั่วโปรเจกต์

**กติกา**: ถ้าฟังก์ชันไม่ต้องอาศัยหน้าตา (ไม่ต้อง return JSX) และมีโอกาสถูกเรียกจากหลายที่ → ใส่ `lib/` ไม่ใช่เขียนแทรกอยู่ในไฟล์หน้าเว็บ

## `store/` — ที่พักข้อมูลกลาง ให้ทุกหน้าเห็นตรงกัน

ปัญหาที่ store แก้: หน้า Home ต้องโชว์ยอดรวมวันนี้, หน้า List ต้องโชว์ทุกรายการ — ถ้าต่างคนต่างไปขอข้อมูลจาก Google Sheet เอง จะช้าและข้อมูลอาจไม่ตรงกัน

`store/expenseStore.ts` เป็นที่เก็บ "รายการรายจ่ายทั้งหมด" ไว้กลางๆ ในความจำของเบราว์เซอร์ — หน้าไหนอยากได้ข้อมูล ก็มาเรียกอันนี้ที่เดียว ไม่ต้องยิง API เอง (ยิง API ให้ store ทำแทน)

ข้อมูลใน store นี้ **หายเมื่อ reload หน้าเว็บ** เพราะเก็บแค่ในความจำ ไม่ได้เก็บลง localStorage — ทุกครั้งที่เปิดหน้าใหม่ต้องขอโหลดข้อมูลจาก Google Sheet ใหม่

## `types/` — นิยามรูปแบบข้อมูล

`types/index.ts` บอกว่า "รายจ่าย 1 รายการ" หน้าตาเป็นยังไง (มี date, item, amount, category, ...) เขียนไว้ที่เดียว ทุกไฟล์ที่ใช้ข้อมูลรายจ่ายจะ import type นี้ไปใช้ กันไม่ให้แต่ละไฟล์นิยามข้อมูลไม่ตรงกันเอง

## สรุปสั้นๆ เวลาจะเขียนโค้ดใหม่

| อยากทำอะไร | ไปเขียนที่ |
|---|---|
| เพิ่มหน้าเว็บใหม่ | โฟลเดอร์ใหม่ใน `app/` |
| เพิ่ม/แก้ endpoint หลังบ้าน | `app/api/.../route.ts` |
| ปุ่ม/การ์ด/ช่องกรอกที่ใช้ซ้ำได้ | `components/ui/` |
| อะไรที่ใช้เฉพาะหน้าใดหน้าหนึ่ง | โฟลเดอร์ย่อยใน `components/` ตามชื่อหน้า |
| ฟังก์ชันคำนวณ/แปลงข้อมูล/เช็คความถูกต้อง | `lib/` |
| ข้อมูลที่หลายหน้าต้องใช้ร่วมกัน | `store/` |
| นิยามรูปแบบข้อมูล (field มีอะไรบ้าง) | `types/` |

## เรื่องเทคนิคที่ควรรู้ก่อนแก้ต่อ

- Next.js เวอร์ชันนี้ค่อนข้างใหม่ มีของเปลี่ยนไปจากที่คุ้นเคยทั่วไป — มีคำเตือนไว้ใน `AGENTS.md` ให้เช็คก่อนเขียนโค้ดที่แตะเรื่อง routing
- ไฟล์เช็ค login ก่อนเข้าเว็บ ชื่อ `proxy.ts` (ปกติโปรเจกต์อื่นมักตั้งชื่อ `middleware.ts`)
- ข้อมูลจริงเก็บใน Google Sheet ไม่ใช่ database — อ่าน/เขียนช้ากว่า database ปกติ ถ้าข้อมูลเยอะมากอาจต้องพิจารณาย้ายออกในอนาคต (ดู README)
- โปรเจกต์นี้ยังไม่มี automated test
